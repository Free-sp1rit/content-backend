3. 后续推进总原则

以项目为主线推进，而不是脱离项目刷概念。
每次只推进一个小模块，必须形成：
能解释 + 能实现 + 能运行 + 能验证。
继续坚持分层边界：
repository 只做数据访问
service 只做业务流程与业务判断
handler 只做 HTTP 输入输出
main 只做依赖装配
默认先让用户自己设计输入输出与骨架，再 review 与补齐。
不要在能力尚未闭环前扩展花哨功能。

4. 新阶段推进顺序（严格按此优先级）

阶段 E：补齐文章编辑闭环
目标：
补上 Content Publishing Backend 中缺失的 EditArticle，完成“创建 -> 编辑 -> 发布 -> 查看”的完整内容流。

必须完成：
EditArticle service
对应 handler
对应路由
作者权限判断
草稿与已发布文章的编辑规则

完成标准：
用户能解释为什么编辑权限属于 service 判断
用户能解释“编辑文章”和“发布文章”为什么是两个业务动作
接口能实际跑通并验证作者身份边界

阶段 F：系统化错误分层
目标：
把当前项目里已经零散存在的错误处理，整理成真正清晰的 repository / service / handler 三层错误模型。

必须完成：
sql.ErrNoRows 的统一翻译策略
业务错误与底层错误区分
常见写接口的 HTTP 状态码映射
重复注册、无权限、文章不存在等错误的统一表达

完成标准：
用户能解释：
哪些错误属于数据库错误
哪些错误属于业务错误
为什么 repository 不直接返回“邮箱已注册”
为什么 handler 只做 HTTP 映射

阶段 G：测试最小闭环
目标：
让项目从“能跑”升级到“可验证”。

必须完成：
至少一组 service 测试
至少一组 handler 或 integration test
至少覆盖一条完整主链路：
登录
创建文章
发布文章
查看详情
或编辑文章

完成标准：
用户知道该测什么，不该测什么
用户能用测试证明关键业务闭环成立
用户开始建立测试分层意识

阶段 H：事务与一致性基础
目标：
让用户理解什么时候需要事务，而不是只会写单表 CRUD。

必须完成：
理解事务的作用与边界
结合当前项目判断哪些操作暂时不需要事务
设计一个需要事务的增强场景，例如：
发布文章 + 写审计日志
创建文章 + 写操作记录

完成标准：
用户能解释为什么“多步写操作”可能需要事务
用户能解释当前已有接口为什么大多暂时不需要事务
用户能说清事务是在 service 组织，而不是散落在 handler

阶段 I：系统保护与运行边界
目标：
在闭环完成后，补最基础的服务保护能力。

推荐内容：
context 超时
请求级超时控制
简单限流
优雅退出
下游资源保护意识

完成标准：
用户能解释为什么请求多不等于都应该同时处理
用户能解释为什么服务端要保护数据库和外部资源
用户开始理解“后端不是只把功能写出来”

阶段 J：项目展示与简历表达
目标：
把项目提升到“可讲、可展示、可用于面试追问”的状态。

必须完成：
README 项目说明
技术栈说明
核心 API 说明
项目结构说明
关键业务与设计取舍说明
一段适合简历的项目描述

完成标准：
用户能在 5 到 10 分钟内讲清项目目标、结构、鉴权链路、文章主流程
用户能说明自己做过哪些设计权衡
项目看起来像一个初期产品，而不是练习代码堆

6. 模型每次指导该如何推进

每次只推进一个模块。
先解释这个模块解决什么问题。
再让用户先设计：
输入
输出
方法签名
分层位置
然后再 review 用户实现。
最后必须补：
运行验证
边界条件
trade-off 解释

禁止：
一次推进过多模块
闭环未成型就扩展花哨功能
用户还没理解边界就直接给整段成品
