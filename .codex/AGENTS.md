# 学习执行原则（新主线约束）

## P1. 项目驱动，但不牺牲理论

从 G2 开始，每一阶段都同时包含两条线：

- **理论线**：理解机制、设计理念、边界条件
- **项目线**：把本阶段知识落到内容发布后端项目中

原则：

```text
学一点理论
→ 做一点项目
→ 回头解释为什么这样设计
```

而不是：

```text
只看资料
或
只堆功能
```

------

## P2. 引导式写代码，不复制粘贴

后续实现遵循：

```text
先定义问题
→ 再设计数据结构 / 接口 / 分层
→ 先写骨架
→ 再补实现
→ 最后复盘
```

具体约束：

- 不直接整段复制成品代码
- 优先自己写 handler / service / repository
- 先写伪代码、函数签名、目录结构，再写具体逻辑
- 每个模块都要能自己解释“为什么这么写”

目标不是“拿到代码”，而是形成：

```text
需求分析 → 数据建模 → 接口设计 → 实现 → 测试 → 复盘
```

这一整套方法论。

------

## P3. 项目范围严格克制

项目以“**可按时完工、可讲清楚、可展示**”为第一原则。

第一版只做核心闭环，不主动扩展复杂功能。
优先级永远是：

```text
完整闭环 > 花哨功能
可解释 > 功能堆叠
```

## G2.1前置，Git工具的基本使用学习

# 一、Git 的核心模型（最重要）

你现在已经建立了 Git 的 **三层结构模型**：

```
Commit History   版本历史
Staging Area     暂存区
Working Tree     工作区
```

关系是：

```
工作区 → add → 暂存区 → commit → 历史
```

也就是：

```
修改代码
↓
git add
↓
git commit
↓
git push
```

# 七、reset（三个级别）

我们重点理解了：

```
git reset --soft
git reset --mixed
git reset --hard
```

区别在于 **影响哪几层**：

| 命令  | commit | 暂存区 | 工作区 |
| ----- | ------ | ------ | ------ |
| soft  | ✔      | ❌      | ❌      |
| mixed | ✔      | ✔      | ❌      |
| hard  | ✔      | ✔      | ✔      |

口诀：

```
soft  后悔 commit
mixed 后悔 commit + add
hard  后悔 commit + add + 修改
```

------

# 八、detached HEAD

你还遇到了一个真实 Git 状态：

```
HEAD detached
```

意思是：

```
HEAD 指向 commit
而不是 branch
```

这种状态下：

```
可以 commit
但 commit 不属于任何分支
```
